# User Stories with Acceptance Criteria — Code Review Assistant

Format: Given/When/Then (Gherkin-style). MVP stories are fully specified; Phase 2/3 stories are lightly scoped since they're backlog, not build targets yet.

---

## Epic 1: Core Review Flow

### Story 1 — Paste a snippet
As a solo developer, I want to paste a code snippet directly into the tool, so that I can get feedback without setting up a repo or CI pipeline.

- **Given** I am on the review page, **when** I paste code into the input area and click "Review," **then** the system accepts the input without requiring a file upload, repo URL, or account.
- **Given** the input area is empty, **when** I click "Review," **then** the system blocks submission and shows a message asking me to paste code first.

### Story 2 — Language support
As a developer, I want the tool to support Python and JavaScript, so that I can use it on the languages I actually write in.

- **Given** I paste valid Python or JavaScript code, **when** I submit it, **then** the system correctly identifies the language and applies the matching review/fallback logic.
- **Given** I paste code in an unsupported language, **when** I submit it, **then** the system tells me clearly which languages are supported rather than returning a silent or malformed result.

### Story 3 — Severity labeling
As a junior developer, I want flagged issues labeled by severity (bug vs. style), so that I know what to fix first.

- **Given** a review returns multiple issues, **when** results are displayed, **then** each issue shows a visible severity tag ("Bug" or "Style").
- **Given** a snippet has both bug-level and style-level issues, **when** I view results, **then** bug-level issues are sorted or visually prioritized above style-level ones.

### Story 4 — Single results panel
As a developer, I want all results shown in one clear panel, so that I'm not hunting across tabs or scrolling through unrelated output.

- **Given** a review completes, **when** results are ready, **then** all issues appear in a single panel without requiring navigation to a separate page or tab.
- **Given** zero issues are found, **when** results are ready, **then** the panel explicitly states no issues were found, rather than appearing blank or broken.

## Epic 2: Explanations

### Story 5 — Plain-language explanations
As a junior developer, I want a plain-language explanation for every flagged bug, so that I understand why it's wrong.

- **Given** an issue is flagged, **when** I view it, **then** it includes a written explanation, not just a rule ID or one-line label.
- **Given** the AI review path is used, **when** an explanation is generated, **then** it references the specific line/construct involved rather than a generic description.

### Story 6 — Jargon-free explanations
As a junior developer, I want explanations without unexplained jargon, so that I don't have to look up terms to understand my own review.

- **Given** an explanation uses a technical term (e.g., "race condition," "mutable default argument"), **when** it's displayed, **then** the term is briefly defined inline rather than assumed as known.

### Story 7 — Consequence-linked explanations
As a self-taught developer, I want explanations that connect a bug to a real consequence, so that I build intuition instead of memorizing fixes.

- **Given** a bug is flagged (e.g., unhandled empty-list case), **when** the explanation is generated, **then** it states what will concretely happen if the code runs as-is (e.g., "this will throw an IndexError when the list is empty"), not just that the pattern is discouraged.

## Epic 3: Reliability & Fallback

### Story 8 — Graceful fallback on AI outage
As a solo freelancer, I want the tool to keep working even when the AI service is down or rate-limited, so that I'm never left with a dead API call.

- **Given** the AI provider returns a timeout, rate-limit error, or quota-exhausted error, **when** this happens, **then** the system automatically retries once and, on continued failure, switches to the static-analysis fallback without the user needing to resubmit.
- **Given** the fallback is triggered, **when** results are shown, **then** the user still receives a complete result set — never an empty screen or raw error message.

### Story 9 — Trustworthy fallback engine
As a developer, I want the fallback to use real static-analysis tools (Pylint/ESLint), so that I can trust results when AI isn't available.

- **Given** the fallback path is active for Python, **when** a review runs, **then** results are generated by Pylint (or equivalent), not a custom regex-only checker.
- **Given** the fallback path is active for JavaScript, **when** a review runs, **then** results are generated by ESLint (or equivalent).

### Story 10 — Clear source labeling
As a developer, I want to clearly see whether a result came from the AI or the fallback, so that I calibrate how much to trust each finding.

- **Given** a review completes via the AI path, **when** results are displayed, **then** a visible, unmissable label reads "AI-reviewed."
- **Given** a review completes via the fallback path, **when** results are displayed, **then** a visible, unmissable label reads "Fallback-reviewed," styled distinctly (not a small footnote).

### Story 11 — Fast fallback response
As a developer, I want the fallback path to return results in a few seconds, so that losing AI access doesn't mean losing my workflow too.

- **Given** the fallback path is triggered, **when** a snippet under 300 lines is submitted, **then** results return within ~3 seconds.

## Epic 4: Trust & Consistency

### Story 12 — Explainable review logic
As a team lead, I want review logic that isn't a total black box, so that I can explain to stakeholders why a bug was or wasn't caught.

- **Given** a result is shown, **when** I inspect it, **then** I can see which engine (AI model or specific linter/rule) produced the finding.

### Story 13 — Consistent output across provider changes
As a team lead, I want consistent output regardless of which AI provider or model is behind the scenes, so that quality doesn't silently drift.

- **Given** the configured LLM provider is swapped (e.g., Gemini to Groq), **when** the same snippet is reviewed, **then** the output format and severity taxonomy remain identical — only the specific findings/explanations may vary.

---

## Epic 5: Privacy & Trust (cross-cutting)

### Story 14 — No code retention
As a developer, I want my pasted code to not be stored or logged after the review, so that I can review proprietary code without a data-retention concern.

- **Given** a review request completes (success or failure), **when** the response is returned, **then** the submitted code is not written to any persistent log or database.
- **Given** I ask what happens to my code, **when** I check the product's stated policy, **then** it matches actual system behavior (no silent logging).

### Story 15 — No account required
As a developer, I want the tool to work without creating an account, so that I can get a quick review with zero friction.

- **Given** I visit the tool for the first time, **when** I want to run a review, **then** I am not prompted to sign up, log in, or provide an email.

## Epic 6: Later Phases (backlog — AC to be finalized when scoped for build)

### Story 16 — Save past reviews *(Phase 2)*
As a developer, I want to save past reviews, so that I can track recurring mistakes over time.

- *(Deferred)* Draft AC: Given a review completes, when I choose to save it, then it persists locally or in a lightweight store and is retrievable in a history view.

### Story 17 — GitHub PR integration *(Phase 3)*
As a team lead, I want the tool integrated into my GitHub PR workflow, so that review happens automatically without manual pasting.

- *(Deferred)* Draft AC: Given a PR is opened on a connected repo, when CI triggers the review, then results post as inline PR comments automatically.
